# 검증 워크스페이스 설정 및 활용 가이드

이 문서는 Persona Lab에서 합성 페르소나를 기반으로 **검증 가능한 가설 워크스페이스**를 만들고, 실제 인터뷰/현장 실험 결과를 기록하는 운영 절차를 설명합니다.

> 중요: 이 앱의 페르소나는 가설을 만드는 데 쓰는 참고 자료입니다. 실제 수요나 구매 의향의 증거가 아니므로, 반드시 인터뷰·관찰·현장 테스트 등 실제 검증 결과로 판단해야 합니다.

## 1. 워크스페이스 개념

검증 워크스페이스는 다음 데이터를 한 묶음으로 저장합니다.

- **세그먼트**: 어떤 페르소나 집단을 볼지 정하는 필터와 표본 크기
- **가설**: 검증할 핵심 주장
- **검증 질문**: 인터뷰나 현장 확인에 사용할 첫 질문
- **검증 결과**: 실제 검증 후 기록하는 응답자 프로필, 요약, 근거 강도, 검증 여부

앱 화면은 세 영역으로 구성됩니다.

| 영역 | 용도 |
| --- | --- |
| 왼쪽 사이드바 | 세그먼트/워크스페이스 목록 선택 |
| 중앙 채팅형 패널 | 선택한 세그먼트의 페르소나 샘플과 검색 결과 확인 |
| 오른쪽 확인 패널 | 선택 세그먼트 정보, 가설/질문, 검증 결과 입력 |

## 2. 서버 실행 설정

### 2.1 기본 설치

```bash
python3 -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt
```

### 2.2 DuckDB 생성

전체 데이터셋을 별도로 내려받은 뒤 DuckDB 메타데이터와 기본 세그먼트를 생성합니다.

```bash
python scripts/create_duckdb.py \
  --dataset-dir ./data \
  --db-path ./db/nemotron_personas_ko.duckdb

python scripts/create_segments.py \
  --db-path ./db/nemotron_personas_ko.duckdb
```

### 2.3 DB 경로 지정

기본 DB 위치가 아닌 파일을 사용할 때는 `PERSONA_LAB_DB`를 지정합니다.

```bash
export PERSONA_LAB_DB=/absolute/path/to/db/nemotron_personas_ko.duckdb
```

### 2.4 웹앱 실행

```bash
python -m uvicorn webapp.app:app --host 0.0.0.0 --port 8787
```

브라우저에서 다음 주소를 엽니다.

```text
http://localhost:8787/
```

원격 서버에서 실행 중이면 서버 IP 또는 터널 주소의 `8787` 포트를 사용합니다.

## 3. 새 검증 워크스페이스 만들기

1. 왼쪽 사이드바에서 `+ New workspace`를 누릅니다.
2. 오른쪽 확인 패널 상단에 새 가설 워크스페이스 폼이 열립니다.
3. 아래 필드를 입력합니다.

| 필드 | 필수 | 설명 | 예시 |
| --- | --- | --- | --- |
| 워크스페이스 이름 | 예 | 왼쪽 목록에 표시될 검증 단위 이름 | `공단 점심 선택 시간 가설` |
| 핵심 가설 | 예 | 실제로 검증할 문장 | `근로자는 출근 직후 오늘 메뉴를 확인하면 점심 선택 시간이 줄어든다.` |
| 첫 검증 질문 | 아니오 | 인터뷰/현장 확인에 바로 쓸 질문 | `오늘 점심 메뉴를 언제, 어디서 확인하나요?` |
| 필터 키워드 | 아니오 | 페르소나 텍스트 검색어. 비워두면 연령/지역만 적용 | `점심`, `공단`, `한식` |
| 지역 | 아니오 | `province` 값과 일치하는 지역명 | `경기`, `서울` |
| 최소/최대 나이 | 예 | 세그먼트 대상 연령 범위 | `20` ~ `64` |

4. 생성 버튼을 누릅니다.
5. 저장이 끝나면 세그먼트 목록이 새로고침되고, 방금 만든 워크스페이스가 자동 선택됩니다.
6. 입력을 취소하려면 취소 버튼을 누릅니다.

## 4. 생성 후 페르소나 확인하기

워크스페이스가 선택되면 중앙 패널에 해당 세그먼트의 페르소나 샘플이 표시됩니다.

추가 탐색이 필요하면 하단 검색창을 사용합니다.

- 검색어 예시: `점심`, `한식`, `교대`, `공단`, `편의점`
- 지역 필터: `전국`, `경기`, `서울`, `인천`, `부산`

검색 결과는 가설을 다듬기 위한 참고 자료입니다. 문장 그대로 실제 고객의 발언처럼 사용하지 마십시오.

## 5. 검증 결과 기록하기

오른쪽 빠른 검증 노트 영역에서 실제 검증 결과를 저장합니다.

1. 가설 선택 드롭다운에서 기록할 가설을 고릅니다.
2. 응답자 프로필 입력칸에 응답자 유형을 적습니다.
   - 예: `30대 제조업 근로자`, `구내식당 이용자`, `현장 관리자`
3. 인터뷰 또는 현장 검증 요약 입력칸에 검증 요약을 적습니다.
   - 실제 개인 식별 정보, 연락처, 회사 내부 정보는 넣지 마십시오.
4. `evidence_level`을 고릅니다.
   - `weak`: 단서 수준
   - `medium`: 반복 패턴 또는 부분 확인
   - `strong`: 여러 명/현장 조건에서 반복 확인
5. 실제로 가설을 지지한다고 판단되면 `validated`를 체크합니다.
6. 저장 버튼을 눌러 저장합니다.

저장된 결과는 다음 API로도 조회할 수 있습니다.

```text
GET /api/validation-results?hypothesis_id=<hypothesis_id>
```

## 6. 운영 기준

### 6.1 좋은 가설 작성법

좋은 가설은 다음 형식을 권장합니다.

```text
[대상 사용자]는 [상황]에서 [문제/욕구]가 있기 때문에 [행동/해결책]을 원할 것이다.
```

예시:

```text
공단 근로자는 출근 직후 당일 메뉴 정보를 빠르게 확인하면 점심 선택에 드는 시간을 줄일 수 있을 것이다.
```

피해야 할 문장:

- `이 서비스는 성공할 것이다.`처럼 검증 행동이 불명확한 문장
- `모든 사용자가 원한다.`처럼 대상이 넓은 문장
- 실제 검증 없이 확정적으로 말하는 문장

### 6.2 검증 질문 작성법

좋은 질문은 사용자의 과거 행동과 현재 불편을 묻습니다.

권장:

- `최근 일주일 동안 점심 메뉴를 확인한 적이 있나요? 어떻게 확인했나요?`
- `메뉴를 몰라서 식당 선택을 바꾼 경험이 있나요?`
- `메뉴 확인에 실패했을 때 어떤 대안을 쓰나요?`

피해야 함:

- `이 앱을 쓰실 건가요?`처럼 예측 의향만 묻는 질문
- `이 기능이 좋죠?`처럼 답을 유도하는 질문

이 저장소는 코드 중심의 공개 가능한 형태를 전제로 합니다. 다음 내용은 커밋하지 마십시오.

## 8. 문제 해결

### 8.1 `+ New workspace`를 눌러도 폼이 안 열림

1. 브라우저 새로고침을 합니다.
2. 서버가 최신 코드로 실행 중인지 확인합니다.
3. 정적 JS에 폼 이벤트가 있는지 확인합니다.

```bash
curl -s http://127.0.0.1:8787/static/app.js | grep workspaceForm
```

### 8.2 브라우저 기본 팝업이 뜸

최신 UI는 `window.prompt`를 사용하지 않습니다. 아래 명령으로 남아 있는지 확인합니다.

```bash
grep -R "window.prompt" webapp/static
```

결과가 나오면 오래된 코드가 실행 중이거나 캐시가 남아 있는 상태입니다. 서버를 재시작하고 브라우저 캐시를 비웁니다.

### 8.3 저장 시 DB 오류가 남

DB 경로와 쓰기 권한을 확인합니다.

```bash
curl http://127.0.0.1:8787/api/health
ls -l "$PERSONA_LAB_DB"
```

`PERSONA_LAB_DB`가 잘못됐거나 읽기 전용 파일이면 저장이 실패합니다.

### 8.4 포트가 이미 사용 중임

```bash
ss -ltnp 'sport = :8787'
```

기존 프로세스를 종료한 뒤 다시 실행합니다.

### 8.5 테스트 실행

```bash
. .venv/bin/activate
pytest -q
```

현재 테스트는 작은 고정 샘플 DB를 사용하므로 전체 데이터셋이 없어도 실행됩니다.

## 9. 운영 체크리스트

새 검증 워크스페이스를 만들 때마다 아래를 확인합니다.

- [ ] 가설이 특정 대상/상황/행동을 포함한다.
- [ ] 첫 검증 질문이 실제 과거 행동을 묻는다.
- [ ] 필터 키워드와 지역이 너무 좁거나 넓지 않다.
- [ ] 합성 페르소나 내용은 참고로만 사용한다.
- [ ] 검증 결과에는 개인정보나 비공개 원문을 넣지 않는다.
- [ ] `weak`, `medium`, `strong` 근거 수준을 보수적으로 기록한다.
- [ ] 검증되지 않은 가설을 확정 사실처럼 문서화하지 않는다.
